# Git 核心與平台 (0.5): Git 與 GitHub 的關係

對於初學者而言，最容易混淆的概念之一就是 Git 與 GitHub。這份文件將專門釐清這兩者的定義與關係。

---

## 核心定義

-   **Git：** 一個**分散式版本控制系統 (Distributed VCS)**。它是一個安裝在您電腦上的**軟體工具**，負責追蹤檔案變更、管理歷史版本、處理分支與合併等核心功能。

-   **GitHub：** 一個**基於 Git 的雲端託管服務平台**。它是一個**網站服務**，提供遠端儲存庫的託管，並圍繞 Git 建立了一整套協作工具。

## 關係比喻：工具與平台

我們可以從一個情境來理解兩者的關係：

假設世界上只有 Git。您可以在自己的電腦上完美地進行版本控制，就像擁有一座功能齊全的本地「倉庫園區」。但問題是，這座倉庫園區只存在於您自己的土地上（您的電腦）。

這時，GitHub 出現了。它提供了一個位於雲端的「中央倉庫園區」。您可以將您本地倉庫的所有歷史紀錄（`push`）同步到這個中央倉庫。

這帶來了兩個革命性的改變：
1.  **跨裝置工作：** 您可以在公司的電腦上 `push` 進度，回到家再從中央倉庫 `pull` 最新的進度下來繼續工作。
2.  **團隊協作：** 您的同事也可以從同一個中央倉庫 `pull` 您的進度，進行修改後再 `push` 回去。這實現了多人圍繞同一個專案進行高效、安全協作的目標。

## 功能對比

| 特性 | Git (工具) | GitHub (平台) |
| :--- | :--- | :--- |
| **主要功能** | 版本控制、歷史追蹤、分支合併 | 程式碼託管、協作、專案管理 |
| **運作地點** | 本機電腦 (Offline) | 雲端 (Online) |
| **核心** | 追蹤程式碼的「內容」與「歷史」 | 提供圍繞程式碼的「社交」與「工作流程」 |
| **代表性功能** | `commit`, `branch`, `merge` | Pull Request, Issues, Actions, Wiki |

## 結論

-   您可以**只使用 Git** 而不使用 GitHub（例如，只在本地進行版本控制）。
-   但**沒有 Git，就沒有 GitHub**。GitHub 的所有功能都建立在 Git 的基礎之上。
-   GitHub 只是眾多 Git 託管服務中的一個，其他知名的平台還包括 **GitLab** 和 **Bitbucket**。
