# Tag & Release - 為專案建立里程碑

當專案開發到一個重要階段，例如準備發布 `v1.0` 版本時，我們需要一個方法來永久地標記這個時間點的「存檔點」(Commit)，方便未來隨時能找到它。這就是「標籤 (Tag)」的用途。

---

## 1. 什麼是標籤 (Tag)？

**白話比喻：一支給歷史存檔點的「永久性書籤」或「里程碑旗幟」。**

-   當您在某個 Commit 上打上一個 Tag (例如 `v1.0.0`)，就等於在那個時間點的程式碼狀態上插了一支旗幟。
-   這支旗幟是固定不動的，它永遠指向那個特定的 Commit。

## 2. 標籤 (Tag) vs. 分支 (Branch)

這兩者很容易混淆，但它們的用途完全不同：

-   **分支 (Branch)**: 像一條**不斷延伸的公路**。它是一個會隨著新的 Commit 而**向前移動**的指標，代表著一條正在進行中的開發故事線。
-   **標籤 (Tag)**: 像公路旁的**一個固定里程碑**。它是一個**固定不動**的指標，錨定在歷史上的某個特定 Commit，代表一個已完成、具有特殊意義的時間點。

| 特性 | 分支 (Branch) | 標籤 (Tag) |
| :--- | :--- | :--- |
| **移動性** | 會移動 | 固定不動 |
| **用途** | 進行中的開發 | 標記已完成的版本 |
| **比喻** | 故事線、開發公路 | 里程碑、版本旗幟 |

---

## 3. 從 Tag 到 Release (GitHub 的應用)

Git 本身只提供了 `tag` 這個標記功能。而 GitHub 在此基礎上，建立了一個更強大的概念：**Release (發布)**。

一個 GitHub Release = **一個 Git Tag** + **詳細的發布說明 (Release Notes)** + **可供下載的附加檔案** (例如安裝檔 `.apk`, `.exe`)。

這使得 Release 成為一個對外正式發布版本的最佳方式。

## 4. 如何應用與追蹤產品版本

一般來說，您不需要親自去打 Tag，但需要知道如何閱讀它。

1.  **找到 Releases 頁面**: 在 GitHub 專案主頁的右側欄，通常會有一個「**Releases**」的區塊。
2.  **閱讀版本資訊**: 點擊進入後，您會看到一個所有已發布版本的列表。每一筆 Release 通常會包含：
    -   **版本號 (Tag name)**: 例如 `v1.0.1`。
    -   **發布日期**: 這個版本是什麼時候發布的。
    -   **發布說明 (Release Notes)**: 這是最重要的部分！開發團隊會在這裡條列出這個版本新增了哪些功能、修復了哪些 Bug。這是您追蹤產品迭代最直接的資訊來源。
    -   **相關的存檔點**: 連結到這個版本對應的程式碼快照。
