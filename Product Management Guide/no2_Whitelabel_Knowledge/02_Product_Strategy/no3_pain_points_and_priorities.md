# 痛點地圖：三大方向與資源分配

## 角色定位

PM 每天面對大量需求，資源不夠全部做完。這份框架讓每個工作項目有明確的業務意義，也幫助判斷哪個方向影響最大。

---

## 三大優先方向

白牌平台的功能規劃，最終都在服務站方的三個核心目標：

```
┌──────────────────────────────────────────┐
│  降低推廣成本  │  提高玩家留存  │ 降低管理成本 │
└──────────────────────────────────────────┘
```

- 這不是三個等權重的方向
- 不同階段、不同站台的狀況，三個方向的優先級不同
- PM 要能判斷現在最應該改善的是哪個方向

---

## 方向一：降低推廣成本

**核心問題：** 花同樣的廣告費，能不能帶來更多真正付費的玩家

對應的是 ACRR 中的 A 獲客和 C 轉換兩個環節。

### 主要痛點

**推廣工具不完整**

- 追蹤代碼沒串好 → 站方不知道哪個廣告有效 → 廣告預算無法優化
- 這是獲客效率的底層問題

**存款頁轉換率低**

玩家好不容易被廣告吸引進來，在存款頁流失了。可能原因：

- 三方通道名稱看不懂，玩家選不到自己熟悉的方式
- 存款頁 UI 不清楚，步驟不直覺
- 沒有顯示對應的優惠，玩家不知道首存有紅利

**提款頁體驗差**

- 很多玩家試試看能不能提款就決定要不要長期留下
- 提款流程卡、審核慢、成功後沒有通知 → 玩家失去信任 → 影響口碑和自然增長

**自然增長機制缺乏**

- 口碑和分享是成本最低的獲客方式
- 目前缺乏讓玩家主動分享的機制，例如 WhatsApp 分享連結，或是分享後能獲得的獎勵

### 功能方向

| 痛點 | 對應功能 |
|------|----------|
| 追蹤碼沒串好 | 廣告追蹤工具串接，Click ID、Meta Pixel |
| 存款頁轉換率低 | 存款頁 UI 優化、三方通道名稱系統化、存款頁優惠推薦 |
| 提款體驗差 | 提款頁面優化，稽核狀態顯示、提款成功通知 |
| 缺自然增長 | 社群分享獎勵、全民代理連結機制 |
| 帳號轉換率低 | Google 和 Facebook 快速登入、手機 Email 雙重選擇 |

---

## 方向二：提高玩家留存

**核心問題：** 玩家來了之後，有沒有理由繼續留著玩

對應的是 ACRR 中的 R 留存和 R 喚回兩個環節。

### 主要痛點

**優惠設定彈性不足**

- 站方想設計各種留存優惠，分眾優惠、階梯式獎勵、排行榜，但平台的優惠功能設定選項有限
- 無法實現站方想要的策略

**VIP 制度缺乏差異化**

- VIP 玩家感受不到差異，頭銜沒有實質意義
- VIP 體驗和普通玩家相同，大課長沒有理由繼續在這裡花大錢

**玩家餘額歸零沒有救援機制**

- 玩家把餘額輸光後直接消失
- 在餘額歸零那一刻推送救援金，喚回率會高很多
- 目前缺乏這個即時觸發機制

**沉睡玩家的觸達工具不足**

- 站方知道哪些玩家沉睡了，靠報表篩選
- 但觸達的工具不齊全，站內信分群、SMS、Email 批量推送都不完整

**遊戲探索體驗差**

- 玩家不知道哪些遊戲有專屬優惠、哪些遊戲是熱門的
- 每次進來都只玩同一個遊戲，沒有探索新遊戲的動機

### 功能方向

| 痛點 | 對應功能 |
|------|----------|
| 優惠彈性不足 | 分眾設定 by VIP、標籤、層級，一鍵領取，優惠種類擴充排行榜、多輪盤 |
| VIP 差異化不足 | VIP 客服連結分層、優惠金額 by VIP 等級、排行榜展示 |
| 餘額歸零沒救援 | 餘額歸零通知、自動觸發救援金 |
| 沉睡玩家觸達 | 站內信分群 by 標籤、電話、VIP，App Push，Email 批量推送 |
| 遊戲探索差 | 遊戲顯示可解鎖優惠、個人化遊戲推薦、熱門遊戲強制置頂 |
| 提款留存 | 提款手續費 by VIP 大課長少付費，USDT 出款 |

## 方向三：降低管理成本

**核心問題：** 完成同樣的工作，有沒有辦法用更少的人力和時間

這個方向的功能不直接影響玩家，但影響站方的運營效率，也影響站方對平台的評價。

### 主要痛點

**報表速度慢、欄位不足**

- 全玩家列表查詢太慢，也缺乏很多篩選條件，例如最後存款時間、近 30 天活躍天數
- 站方要做玩家分析時，常常要等很久，或是要手動匯出再用 Excel 處理

**金流審核效率低**

- 存款單看不到這張單等了多久
- 沒有自動催單機制，超過一定時間未到帳要提醒
- 提款審核要一張一張手動確認，沒有自動放行條件設定

**後台操作缺乏批量功能**

- 幾乎所有後台功能都只能一個一個操作
- 缺乏批量開關帳戶、批量設定優惠、批量調整代理設定

**代理管理功能分散**

- 代理後台和主後台的功能設計不一致，PC 和 H5 長得不一樣
- 代理自己的管理工具也不齊全，缺乏代理後台客服功能

### 功能方向

| 痛點 | 對應功能 |
|------|----------|
| 報表太慢 | 查詢 API 優化用 Elastic Search，合併報表頁面，增加篩選條件 |
| 金流審核效率 | 存款催單機制、花費時間顯示、失敗提醒、自動出款條件 |
| 批量操作缺乏 | 批量啟用停用帳戶、批量轉線、批量人工調帳 |
| 代理管理分散 | 代理後台客服、代理業績獎金管理、AG PC 和 H5 統一介面 |
| 會員管理 | 人工驗證工具、資料留存期限設定 |

## 資源分配的判斷框架

有了三個方向，還需要在同一個方向內決定功能的優先順序。

### ACRR 哪個階段

- 如果站方現在獲客成本很高，優先做降低推廣成本的功能
- 如果獲客沒問題但玩家很快流失，優先做留存功能
- 不同站台可能處於不同的 ACRR 階段

---

### 功能的性質

| 性質 | 描述 | 優先考量 |
|------|------|----------|
| 新功能 | 完全沒有的功能 | 業務增量是否明確 |
| 邏輯漏洞 | 現有功能有 bug 或設計缺陷 | 影響多少玩家和站台 |
| 體驗差 | 功能存在但使用體驗很差 | 修好能減少多少客服量 |
| 做一半的做完 | 有功能但不完整，導致另一端做白工 | 這個最優先，因為已有成本 |

---

### 覆蓋面

- 服務所有站台的通用需求，優先級遠高於只服務單一站台的特殊需求

---

### 做一半的事情最優先

已投入開發成本、但因某個環節未完成而無法使用，是最浪費的狀態，要最優先補完。

例子：

- 廣告追蹤代碼後端已經可以接收，但前端沒有串接，後端的工作白費
- 吉祥物功能做了，但優惠設定沒有抓到吉祥物的觸發條件，玩家抓吉祥物但沒有任何獎勵，功能做白工

---

## 常見誤解

**三個方向都很重要，要同時推進**

- 三個方向同時推進 → 每個方向都做了一點但都沒到位
- 先判斷瓶頸在哪裡，集中資源突破一個方向，再移動到下一個

**後台功能不是玩家看得到的，優先級低**

- 後台難用 → 操作人員花兩倍時間在重複性工作上 → 站方對平台滿意度下降
- 降低管理成本和降低推廣成本一樣重要
