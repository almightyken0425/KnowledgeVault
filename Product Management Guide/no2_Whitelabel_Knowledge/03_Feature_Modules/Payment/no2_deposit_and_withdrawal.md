# 存款與提款

## 角色定位

存款和提款直接決定玩家信任感和站方的運營成本。理解每一種通道的完整流程，才能設計出站方真正需要的功能。

---

## 存款流程：一般法幣

### 公司帳戶 / 電子錢包，Offline 型

**前置作業，站方執行**

- 在外部申請銀行帳戶或電子錢包帳號
- 在後台設定帳戶資訊，包含銀行名稱、帳號、帳戶姓名
- 設定此帳號的最低最高存款金額和費率

**玩家存款流程**

- 玩家選擇主通道
- 玩家選擇收款帳號
- 玩家輸入存款金額
- 玩家依顯示的帳號自行轉帳
- 玩家提交存款資訊，包含轉帳金額、時間、末五碼
- 系統建立存款單
- 站方人工確認收款
- 系統上分

**日常維護，站方執行**

- 核對帳戶收款是否與存款單一致
- 監控帳戶使用頻率，避免單一帳戶收款頻繁成為死戶
- 多帳戶輪訓管理，定期切換主要收款帳戶

---

### 三方帳戶，Online 型

**前置作業，站方執行**

- 向第三方支付商申請商戶帳號
- 在後台設定商戶資訊，包含 API Key、商戶代碼
- 設定 Bank Mapping，指定此三方支援哪些銀行

**玩家存款流程**

- 玩家選擇主通道
- 玩家選擇子通道，也就是三方通道
- 玩家輸入存款金額
- 玩家提交存款資訊
- 系統建立存款單
- 玩家依三方介面完成轉帳
- 三方系統自動回調平台確認收款
- 系統上分

---

## 存款流程：虛擬貨幣

### 公司錢包，Offline 虛擬幣型

**前置作業，站方執行**

- 在站外申請虛擬錢包
- 在後台配置錢包資訊，包含幣別、地址
- 設定匯率和扣款比例

**玩家存款流程**

- 玩家選擇主通道幣別，如 USDT
- 玩家選擇子通道，也就是對應的虛擬錢包
- 系統顯示 QR Code 和當前匯率
- 玩家依帳號轉帳
- 玩家提交存款資訊
- 系統建立存款單
- 站方確認收款
- 系統依匯率換算後上分
- 站方在站外將小幣匯兌為 USDT

**日常維護，站方執行**

- 比對平台後台和錢包後台的收款是否一致
- 核對三方錢包的付費記錄
- 定期更新匯率設定

---

### 三方錢包，Online 虛擬幣型

**前置作業，站方執行**

- 向第三方虛擬幣支付商申請商戶
- 配置商戶資訊和匯率扣款比例

**玩家存款流程**

- 玩家選擇主通道幣別
- 玩家選擇子通道，也就是三方錢包
- 系統顯示 QR Code 和匯率
- 玩家依帳號轉帳
- 三方自動回調平台
- 系統確認收款
- 系統依匯率上分
- 三方在站外完成小幣匯兌 USDT

---

## 匯率設定的複雜度

虛擬幣金流需要解決兩個層次的匯率問題。

**USDT 對法幣匯率**

- 手動設定：站方自行填入匯率，彈性高但需要人工維護
- API 自動更新：接市場報價，準確但整合複雜

**其他幣別對 USDT 匯率**

- 非穩定幣的匯率波動大，風險高
- 常見做法是不開放非穩定幣，只做 USDT

**推薦解法**

```
USDT 對基準法幣
├── 手動匯率：站方自填
├── API 匯率：自動更新
└── 匯差設定：固定金額 或 百分比

其他幣別 對 USDT
└── 各自設定匯率和匯差
```

## 存款後台功能

### 存款管理

站方操作人員每天面對大量存款單，後台需要提供以下功能：

| 功能 | 說明 |
|------|------|
| 催單提醒 | 某張存款單超過一定時間未處理時主動提醒 |
| 花費時間顯示 | 每張存款單從建立到現在花了多久 |
| 失敗提醒 | 存款失敗自動通知站方處理 |
| 審核過久警示 | 人工審核超過閾值自動標記 |

### 帳戶管理

- 帳戶排序管理
  - 存款帳戶：餘額少的排前面，餘額多的排後面，確保帳戶餘額均勻使用
  - 提款帳戶：餘額多的排前面，餘額少的排後面，優先使用有錢的帳戶
- 批量啟用停用：一鍵切換多個帳戶的狀態，通道出問題時快速處理

## 提款流程

**玩家發起提款**

- 玩家填入提款金額和提款帳戶資訊
- 系統驗證稽核是否通過
  - 未通過稽核直接在提款頁面阻擋，不讓玩家送出
- 系統建立提款單

**站方審核**

- 風控自動檢查：IP、FP、銀行帳戶黑名單
- 人工審核確認
  - 後台要能顯示玩家的提款 QR Code
  - 要能一鍵複製玩家姓名，對應銀行轉帳
- 自動出款
  - 低風險玩家可設定直接自動放行，不需要人工審核

**完成出款**

- 系統更新提款單狀態
- 玩家收到提款成功通知

### 提款後台功能

| 功能 | 說明 |
|------|------|
| 提款審核資訊 | 顯示玩家 QR Code、快速複製姓名 |
| 自動出款條件 | 依風控狀態自動放行低風險提款單 |
| 提款上下限分設 | 代理和玩家的提款上下限可以分開設定 |
| 批量扣款調帳 | 人工批量對多個玩家執行不同金額的加扣款 |
| 人工添加稽核 | 對特定提款單手動補加稽核要求 |

## 站方常見金流情境

**情境一：通道突然失效**

```
某三方通道在下午開始出現大量失敗
站方需要：
- 立刻調低這個通道的優先級
- 讓玩家優先選到其他通道
- 通知三方廠商處理
- 通道恢復後再調回來
```

後台需要支援：快速調整通道優先級、批量啟用停用帳戶

**情境二：特定通道額度快滿**

```
某個三方通道今天的額度只剩不多
站方需要：
- 讓這個通道的存款優先進來，把額度刷完
- 可以透過設定特定金流優惠、預設通道、加上標籤
```

後台需要支援：通道優先設定、金流相關優惠觸發

**情境三：銀行帳戶收款過多**

```
某個銀行帳戶這週收款太頻繁
站方需要：
- 切換到其他帳戶輪流收款
- 避免這個帳戶被銀行標記成死戶
```

後台需要支援：多帳戶輪訓、批量切換帳戶狀態

## 常見誤解

**存款失敗是技術問題，PM 管不了**

- 存款失敗有技術原因，也有設計原因
- 通道顯示不清楚、狀態回饋不即時，都是 PM 可以改善的
- 降低存款失敗率是 PM 直接能貢獻的業務指標

**提款越快放行越好**

- 提款速度和風控之間有直接的取捨關係
- 放行越快 → 套利窗口越短 → 套利成本越低
- 自動出款條件的設計，要在速度和安全性之間找到平衡點
