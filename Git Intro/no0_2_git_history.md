# Git 的故事 緣起與維護

了解一個工具的歷史,有助於理解它的設計哲學。

---

## 創造者: Linus Torvalds

Git 的創造者是 Linus Torvalds,他同時也是 Linux 作業系統核心的創造者。

在 2005 年,開發 Linux 核心的團隊原本使用一個名為 BitKeeper 的商業版本控制系統。但後來因為授權問題,他們無法再免費使用。當時市面上沒有任何一個版本控制系統能滿足 Linux 核心這樣龐大、分散式開發的需求。

為了解決這個問題,Linus Torvalds 花了大約兩週的時間,用 C 語言寫出了 Git 的原型。其設計目標為: 速度快、設計簡單、支援非線性開發即數千個平行的分支、完全分散式、能有效處理像 Linux 核心這樣的大型專案。

---

## 維護者

不久之後,Linus Torvalds 將 Git 的維護工作交給了另一位傑出的開發者 Junio Hamano。從 2005 年至今,Junio Hamano 一直是 Git 專案的核心維護者,帶領著全球的開源社群持續改進和維護 Git。

### 開源社群貢獻與治理

Git 的開源社群並非一個實體組織,而是指所有透過公開郵件列表 Mailing List 參與討論、回報問題、以及貢獻程式碼的開發者們。

那麼,是否任何人都可以修改 Git 的原始碼? 答案是可以提案,但不能直接修改。這套治理模式是防止混亂的關鍵:

- **提案 Patch:** 任何開發者都可以針對 Git 的功能或錯誤修復,準備一份程式碼變更,稱為修補檔 Patch。
- **公開審查 Public Review:** 開發者將修補檔提交到 Git 的郵件列表,接受來自全球社群成員的公開審查。任何人都可以針對這份程式碼提出意見、質疑或改進建議。
- **維護者合併 Maintainer Merge:** 只有在經過了嚴格的社群審查,並獲得核心維護者 Junio Hamano 或其信任的子系統維護者的認可後,這份修補檔最終才會被合併到官方的 Git 原始碼中。

這個流程確保了每一行進入 Git 的程式碼都經過了多方檢驗,兼顧了開放性任何人都能貢獻與品質控制並非任何程式碼都能進入,從而避免了混亂。

## 維護模式: 混合式的開源生態

Git 的持續發展與維護,並非單純依賴志工的無償貢獻,而是建立在一個由個人、企業與非營利組織共同構成的混合式生態系統之上。

### 企業贊助

許多深度依賴 Git 的科技公司會直接投入資源以確保其穩定性。這是一種策略性投資,因為它們的業務與 Git 的健康發展息息相關。

- **Google:** Git 的核心維護者 Junio Hamano 自 2010 年以來即受僱於 Google,全職負責 Git 的開發與維護工作。
- **Microsoft:** 作為全球最大 Git 平台 GitHub 的擁有者,Microsoft 也是主要貢獻者。他們不僅僱用開發者為 Git 核心貢獻程式碼,特別是在效能方面,也透過 GitHub 推動整個生態系的發展。

### 非營利組織支持

- **Software Freedom Conservancy SFC:** Git 專案的法律和財務歸屬在 SFC 這個非營利組織之下。SFC 為 Git 提供法律顧問、財務管理和行政支持,確保專案能獨立運作,並保護其開源精神。
