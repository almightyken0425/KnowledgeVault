# 競品研究：OKVIP

## 角色定位

研究競品的功能設計和運營模式，能幫助理解業界標準，以及站方對平台功能的期待從何而來。

---

## OKVIP 的組織架構

OKVIP 採用多地區多功能的組織架構：

**地區分工**

- 馬來：回訪中心，透過電銷團隊維護高價值玩家關係
- 越南：主要營銷團隊，管理優惠中心的前後台和主要官網前台
- 泰國：獨立的在地化營銷團隊

**技術分工**

- 優惠中心：由 Vin 負責，包含前台和後台邏輯
- 官網前台：OKVIP 自己的前台，由 Vin 協助提供邏輯
- CRM：匯集多個包網的玩家資料，用於大數據分析，由勁寶使用

## OKVIP 的優惠系統設計

### 後台設定邏輯

- 優惠列表中展開每個優惠的詳細設定
- 後台沒有優惠類型的下拉選項，每個優惠都是一個設定模板
- 開發人員基於模板新增選項，而非操作人員自己設計
- 這意味著 OKVIP 的優惠彈性較低，但設定門檻也較低

### 優惠申請和派發流程

- 前台文宣透過外部系統提供，玩家在前台申請
- 系統審核流程依序包含申請驗證和優惠邏輯驗證
- 系統自己有採購設備指紋 FP 服務，用於識別多帳號
- 審核頁面顯示同姓名的註冊數量和同 IP 的註冊數量，輔助人工審核

### 派發機制

- 自動派發為主，大多數優惠在條件達成後自動發放
- 需要人工確認的案例才進入手動審核頁面

## OKVIP 的特色功能

### TG Bot 客服工具

- 使用 Telegram Bot 讓客服人員可以快速查詢玩家申請優惠失敗的原因
- 前台不顯示失敗原因，避免玩家針對系統規則進行規避
- 客服拿到失敗原因後，可以選擇性告知玩家

### 銀行白名單

- 玩家的提款銀行帳號需要是在白名單內的銀行
- 確保提款只流向站方能夠識別和控管的銀行體系

### 資料留存設計

- 黑名單用途的資料長期保存，不因玩家帳號刪除而清除
- 姓名、IP、FP 等黑名單資料持續累積

### 測試和 Debug 工具

- 後台有內部人員測試工具，可以查看申請不通過的詳細原因，格式為 JSON
- 前台對玩家不顯示錯誤訊息，避免規則被破解

## OKVIP 的直播平台整合

OKVIP 與直播平台有整合設計：

- 合作直播平台包含 MB66 和 GPK
- OKVIP 負責前台顯示，GPK 提供後端 API
- 玩家透過主錢包扣款進行直播打賞
- 透過落地頁導流讓玩家從直播進入博弈平台

## OKVIP 的優惠類型參考

OKVIP 使用中的特殊優惠類型，可作為功能規劃的參考：

**特定輸法**

```
參與百家樂下注達指定積分，若發生特定結果，平台退還一定比例。
```

需要系統識別特定的牌面組合或遊戲結果。

**社群分享**

- 玩家在 Facebook、Telegram、TikTok 等平台按照要求更新頭像和發文
- 文章需要達到指定的讚數和評論數才算完成任務
- 需要在多個群組中分享

需要人工審核，系統無法自動驗證社群分享是否符合條件。

**實體禮物**

- 玩家完成條件後可以選擇兌換實體商品，例如電器或飲料
- 涉及庫存管理和實體出貨流程
- 後台需要能追蹤哪個玩家選了哪個商品以及出貨狀態

**限定時間存款**

```
特定時間段內存款的玩家獲得額外獎勵，用於刺激站方希望推動的時間段的存款量。
```

需要系統識別存款的時間點，只對符合時間條件的存款觸發獎勵。

## 從競品研究得到的設計啟發

**系統應該對玩家隱藏失敗原因**

- 顯示詳細的失敗原因等同於告訴玩家如何規避系統
- 客服需要另外的工具查詢失敗原因，提供給玩家時由客服判斷揭露程度

**FP 是必要的基礎建設**

- OKVIP 自行採購 FP 服務，說明這是對業務有明確價值的工具
- 沒有 FP，多帳號識別能力大幅下降

**電銷和回訪中心的功能需求**

- 電銷團隊需要快速取得玩家的基本資訊和行為摘要
- 批次查詢多個玩家、設定查詢時間範圍是核心需求
- 批次查詢範圍通常是最近幾小時到幾天的新玩家

## 與自身平台的差距分析

根據競品研究，識別自身平台可以借鑒的方向：

- 優惠系統的彈性設定，讓站方操作人員能自己設計優惠邏輯而不需要每次都等 RD
- TG Bot 類型的客服輔助工具，提升客服效率
- 跨系統的玩家資料整合，讓電銷和回訪工作有數據支撐
- 黑名單的跨站台共享機制，讓一個站台發現的獎金獵人資訊能保護其他站台
