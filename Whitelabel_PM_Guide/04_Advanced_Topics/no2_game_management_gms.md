# 遊戲商管理系統 GMS：架構設計與建設歷程

## 角色定位

GMS 是 Game Management System 的縮寫，是白牌平台管理多個遊戲商合作關係的核心基礎設施，理解 GMS 的建設邏輯，有助於 PM 理解遊戲商接入、成本拆分和帳務對帳這幾個跨站台的複雜問題。

---

## 問題的起點：單一商戶架構的限制

早期的白牌平台通常用單一商戶帳號接遊戲商，所有站台共用同一個 Key，這帶來幾個根本性的問題：

- 成本不透明：所有站台的流水混在一起，無法獨立計算每個站台的遊戲成本
- 開站瓶頸：每次要為新站台接入遊戲商，都需要 RD 手動操作，速度慢
- 對帳複雜：多個站台的帳務混在一起，財務對帳耗時費力
- 權限混亂：所有站台使用同一個商戶帳號，無法對不同站台設定不同限制

---

## GMS 的解決方案

GMS 透過建立一個獨立的遊戲商管理後台，解決上述問題：

**商戶層次架構**

- 總公司層：擁有遊戲商的直連 Key，掌握整體成本
- 白牌商戶層：GMS 為每個白牌分配獨立的商戶代碼
- 站台層：每個站台在白牌商戶下，可以有自己的限紅設定

**核心目標**

- 客戶遊戲成本獨立：每個站台有自己的 Key，成本可以分開計算
- 開站速度提升：AM 可以在後台直接開商戶和產 Key，不需要 RD 介入
- 對帳速度提升：總後台統一顯示所有商戶的報表

---

## 系統架構設計

### 兩個主要組件

**MPS**

- 直接持有遊戲商的直連 Key
- 向下提供給 GGW 使用
- 提供總體報表視圖

**GGW**

- Game Gateway，遊戲閘道器
- 管理商戶帳號和 Key 的 CRUD
- 對下連接各個白牌站台
- 提供各商戶的報表

---

## 開站流程

新站台接入遊戲商的完整流程：

- 建立總公司帳號
- 遊戲準備
  - 向遊戲商申請商戶號，分直連戶和代理號兩種
  - 取得串接文件、支援幣別、API Key
  - 完成遊戲商串接
- 建立商戶，在 GGW 後台操作
- 登入商戶，開始設定
- 設定限紅，不同站台可以有不同的投注上下限設定
- 帳務對帳

---

## GMS 後台功能清單

### 系統管理

- 遊戲商金鑰管理，CRUD
- IP 加白設定，限制只有指定 IP 才能訪問
- 遊戲上架和下架控制
- 支援貨幣設定

### 商戶管理

- 開商戶
- 限紅設定，不同商戶可以設定不同的投注上下限
- 轉線，將玩家從一個遊戲商商戶轉移到另一個
- 子帳號和權限管理

### 安全登入

- IP 加白
- 二步驗證 2FA
- reCAPTCHA

### 報表與對帳

- 輸贏報表，各商戶的輸贏統計
- 調整細目報表，手動調整紀錄
- 玩家交易紀錄，單筆查詢和批量查詢
- 遊戲報表，總結和每日明細
- 貨幣轉換，多幣別站台的匯率設定
- 捕撈報表，捕魚類遊戲的特殊報表需求

### 操作紀錄

- 操作紀錄，誰在什麼時候做了什麼操作
- 轉帳紀錄
- 打賞紀錄，針對直播功能

## 建設階段規劃

GMS 的建設分兩個主要階段：

**階段一：AM 可以在後台產 Key**

- 目標：讓客戶成功部門的 AM 不依賴 RD，自己在後台操作開商戶
- 功能範圍：建立商戶、遊戲金鑰管理、商戶列表、登入入口、後台首頁、IP 管理
- 此階段完成後，開站速度大幅提升

**階段二：總後台報表**

- 目標：提供 MPS 和 GGW 的統一報表視圖，含權限控管
- 功能範圍：跨商戶的合計報表、不同角色看不同維度的數據

---

## PM 需要理解的商戶類型差異

**直連戶**

- 直接與遊戲商簽約，有自己的 API Key
- 成本結構清楚，遊戲商直接對站台計費
- 適合大型站台，流水量足以支撐直接談判

**代理號**

- 透過白牌平台作為中間商接入遊戲商
- 不需要直接與遊戲商談合約，開站門檻低
- 成本由白牌平台統一結算後再分攤給各站台

---

## 對帳的重要性

GMS 的帳務對帳功能對站方的日常運營非常關鍵：

- 遊戲商的報表和平台報表的數字要能對得上
- 數字對不上時，要有工具能快速找到差異在哪一筆
- 調整類的操作，例如補分或扣分，要有獨立的細目報表記錄

沒有完善對帳工具的站台，財務人員每個月要花大量時間手動核對，是高度依賴人力的重複性工作。
