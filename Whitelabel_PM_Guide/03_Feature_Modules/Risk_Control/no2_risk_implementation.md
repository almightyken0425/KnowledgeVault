# 風控實作

## 角色定位

風控是站方操作人員每天執行的流程，理解這些日常工作，才能設計出真正降低人工負擔的後台功能。

---

## 優惠審核的風控流程

風控人員審核優惠申請時的工作流程：

- 查看申請列表，找出等待審核的申請單
- 對可疑申請展開調查
  - 查詢申請玩家的 IP 和 FP 關聯帳號
  - 確認關聯帳號中有多少個也申請了同一個優惠
  - 查看玩家的歷史存款和提款行為是否符合正常模式
- 對確認可疑的申請執行處理
  - 拒絕該次申請
  - 對玩家帳號加上風控標籤，影響後續申請資格
  - 批量對關聯帳號進行相同處理

**後台需要支援**

- 申請列表要顯示風控關鍵信號，不需要點進每一筆才能看到
- FP 和 IP 的關聯帳號數要直接在申請列表顯示
- 批量處理：選取多筆申請，一次拒絕並批量加標籤

---

## 提款審核的風控流程

提款審核時，風控人員確認的重點：

- IP 和 FP 關聯：這個提款帳號與多少個其他帳號有關聯
- 銀行帳戶狀態：提款目標銀行帳號是否在黑名單中
- 稽核通過確認：玩家是否確實完成了有效流水，而不是用對沖投注洗完
- 提款時機：玩家是否在拿到優惠後立刻提款，沒有其他遊玩行為

**自動出款條件**

- 對低風險玩家設定自動放行，不需要人工審核
- 觸發自動出款的條件通常包含：風險指數低於閾值、歷史存提款記錄正常、帳號註冊時間超過一定天數

## 流水管理中的風控設計

流水計算是風控中最容易被玩家鑽漏洞的環節：

**對沖投注偵測**

- 玩家在同一局同時押莊和閒，兩注互相對消，無論結果如何都能洗完流水
- 系統要能識別同一玩家在同一時間對同一局的雙向投注
- 識別後可選擇排除這些注單不計入有效流水

**排除遊戲設定**

- 賠率接近 1 的遊戲，玩家可以用極低的風險快速洗流水
- 例如某些骰寶押大小、百家樂押和局等特定玩法
- 站方要能自由設定哪些遊戲不計入優惠稽核流水

**Excluded Game 顯示**

- 玩家需要知道哪些遊戲不算稽核流水，避免不滿情緒
- 前台要能清楚顯示每個優惠的排除遊戲清單
- 在遊戲列表頁，要能顯示哪些遊戲對當前優惠不計入流水

## 黑名單維護流程

黑名單需要持續維護，不是一次設定就不動：

**加入黑名單的觸發點**

- 風控人員確認某個銀行帳號是獎金獵人使用的，立即加入黑名單
- 系統自動偵測：某個 IP 在短時間內出現大量新帳號，自動標記
- 跨站台共享：業界已知的獎金獵人名單可選擇性採用

**黑名單的層次**

- 硬性拒絕：命中黑名單直接拒絕提款或申請
- 軟性警示：命中黑名單但不直接拒絕，只提示審核人員需要特別注意
- 不同類型的黑名單應對應不同的處理層次

---

## 後台操作紀錄的設計要點

- 紀錄粒度：要細到具體操作內容，例如將玩家 A 的 VIP 從 3 改為 5，而不是只記錄更新 VIP
- 可查詢維度：要能用玩家帳號查誰改過這個玩家、也要能用操作人員帳號查這個人做過什麼
- 時間精度：紀錄到秒，方便對照其他系統記錄
- 不可修改：操作紀錄本身不能被任何角色刪除或修改

---

## 風控功能的版本規劃邏輯

風控功能有明確的建設順序：

- 先建立報表，有數據才能看到誰是可疑玩家
- 再建立查詢工具，FP 和 IP 關聯查詢讓人工調查效率提升
- 再建立指數，有足夠的行為數據後，才能計算出有意義的風險分數
- 最後才能做自動化風控，指數穩定可信後，才讓系統自動處理低風險案例

跳過前面階段直接做風險指數，分數沒有依據，反而讓站方失去信心。

## 常見誤解

**風控越嚴越好**

- 過嚴的風控誤傷正常玩家，大課長被擋提款就離開
- 風控的目標是精準，不是嚴格

**FP 可以完全防止多帳號**

- FP 技術可以被規避，例如使用無痕模式、更換瀏覽器、使用不同設備
- FP 是重要信號，但不能作為唯一判斷依據
- 要結合多個信號來判斷，IP、姓名、銀行帳號、行為模式等綜合考量
