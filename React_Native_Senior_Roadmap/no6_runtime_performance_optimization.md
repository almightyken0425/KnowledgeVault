# React 效能優化與進階記憶機制

## 機制定位
React 的預設行為是當元件的狀態或收到的屬性發生改變時，會觸發重新渲染。這個機制確保了畫面隨時與資料保持同步。
然而在複雜的應用程式中，某些運算可能非常耗時，或者某些展示區塊異常龐大。如果每次資料更新都導致這些龐大的結構重新計算與渲染，將會造成嚴重的效能瓶頸。
效能優化的核心目標就是打破這個預設行為，透過記憶體空間換取運算時間，精準控制哪些部分可以被重複使用，哪些部分必須重新執行。

---

## 核心概念

### 記憶化本質
記憶化的概念相當於設立一個快取檢查哨。
每次執行任務前，先檢查輸入的依賴條件是否發生改變。
若依賴條件維持原狀，則直接從記憶庫調出前次運算好的結果。
若依賴條件已改變，才重新放行進行完整運算。

### 依賴陣列
依賴陣列是記憶機制的靈魂核心。
它負責告訴 React 到底要監視哪些變數。
只要清單中的任何一個變數與前一次渲染時不同，React 就會判定記憶失效，進而重新計算或是重新綁定參考。

## 運作邏輯

### 記憶耗時運算結果 useMemo
**機制定位:** 針對單一組件中的重量級運算進行記憶。
**核心特徵:** 回傳一個純值，只有當依賴陣列的變數改變時才會重新執行該運算。
**使用者視角:** 開發者將一段巨大迴圈或是複雜資料過濾邏輯包裝起來，確保在無關狀態更新時不被波及。
**行為:**
- 將高耗時邏輯寫在傳入的首位參數中
- 將牽涉該運算的狀態變數放入依賴陣列
- 將回傳的值指派給常數供渲染使用

### 穩定函式參考 useCallback
**機制定位:** 針對傳遞給下層接收方的事件處理器進行記憶。
**核心特徵:** 回傳一個受保護的函式宣告。
**背後原理:** 每次渲染時宣告的函式即使內容相同，記憶體位址也完全不同。透過新機制可以將此記憶體位址鎖定，除非外在依賴改變否則不會重新宣告新位址。
**行為:**
- 將事件處理邏輯寫在傳入的函式中
- 將該邏輯內有使用到的狀態變數放入依賴陣列
- 將這份穩定參考的函式作為屬性傳遞給下層顯示區塊

### 阻擋無謂渲染 React.memo
**機制定位:** 對整個接收方顯示區塊進行封裝保護。
**核心特徵:** 該高階函數包裝過後會自動比較前後次收到的屬性是否相同，相同則強制跳過渲染。
**對比分析:** 前述兩種機制是從提供方出發保護資料流出，本機制則是對接收方升起防護罩拒絕無效資源消耗。這些機制必須完美搭配發生效用。若提供方沒有穩定好屬性參考就隨意派發，這層防護罩將無法發揮任何作用。

## 實際案例

### 過濾極巨量商品列表
當畫面同時具備商品清單與深淺色主題切換功能時。
如果不對過濾商品的邏輯施加保護，使用者單純切換深淺色外觀時，應用程式依然會把上萬筆商品重新篩選一次，造成操作時發生不合理的卡頓。
運用數值記憶機制將過濾後的清單妥善保存，切換主題的動作將會瞬間完成。

### 避免顯示區塊因為單一輸入發生波及
當畫面上半部是一個簡單的 `文字輸入框`，下半部是包含數百個節點的複雜圖表時。
使用者在 `文字輸入框` 打字的每一次敲擊都會觸發整體狀態更新。
若不使用防護機制將圖表包裝起來，每一個字母輸入都會感受到明顯延遲。
將圖表區塊保護處理後，由於傳給圖表的資料未曾發生變化，打字時下方的巨型圖表將完全靜止不受到任何波及。
